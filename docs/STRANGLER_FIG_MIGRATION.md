# Strangler Fig Migration

이 문서는 SIMVEX가 해커톤 MVP에서 출발해 4-layer 클린 아키텍처로 도메인을 어떻게 이관해 왔는지를 기록한 진행 일지입니다.
*결과*가 아니라 *과정*을 남기는 이유는, 같은 규모의 리라이트를 다시 한다면 무엇을 그대로 두고 무엇을 바꿔야 할지를 판단할 수 있는 근거가 필요하기 때문입니다.

<br />

## 전략 요약

Strangler Fig는 레거시 시스템을 한 번에 갈아엎지 않고, **새 구현이 도메인 단위로 레거시를 흡수**하도록 만드는 마이그레이션 패턴입니다. SIMVEX는 이 패턴을 다음 세 가지 규칙으로 운영했습니다.

- 신규 코드와 레거시 코드는 **같은 저장소에서 공존**함 (`apps/api` ↔ `legacy-ui/simvex-ui-main`).
- 한 도메인이 4-layer를 완전히 통과하기 전에는 다음 도메인으로 넘어가지 않음.
- 각 도메인의 이관이 끝나면 단위 + 통합 테스트가 그린이어야 함.

이 세 가지 규칙은 *속도*보다 *완결성*을 우선합니다. 한 도메인을 어중간하게 옮기면, 다른 도메인이 따라갈 기준이 사라지기 때문입니다.

## 도메인별 이관 기록

### 1. `memos` — 기준 사례 확립

가장 단순한 CRUD 도메인이었기 때문에 가장 먼저 옮겼습니다. 목적은 *기능을 옮기는 것*이 아니라 **다른 도메인이 따라갈 4-layer 템플릿을 만드는 것**이었습니다.

이 과정에서 Route 스키마 → Controller 매핑 → Service 순수 함수 → Repository 인터페이스로 이어지는 한 묶음이 코드 단위로 굳어졌고, 이후 모든 도메인은 이 묶음을 복제해서 시작했습니다.

### 2. `workflow / Node · Connection` — 도메인 분리의 첫 시험대

워크플로우는 단일 엔티티가 아니라 *노드(Node)*와 *연결(Connection)*이라는 두 개의 도메인 객체로 구성됩니다. 두 객체를 하나의 도메인으로 묶을지, 분리할지를 결정해야 했습니다.

분리한 이유는 두 객체가 **다른 라이프사이클**을 가지기 때문입니다. 노드는 단독으로 존재할 수 있지만, 연결은 항상 두 노드 사이에만 존재합니다. 두 객체를 한 Service에 묶으면 노드 삭제 로직에 연결 삭제 부수효과가 박혀버리는 결합이 생깁니다.

### 3. `ai` — 외부 의존성 격리

AI 도메인은 외부 LLM API에 의존합니다. 이 의존성이 Service에 박히면 단위 테스트가 외부 호출 없이 불가능해집니다.

해결책은 LLM 호출을 별도 인터페이스 뒤로 빼는 것이었습니다. Service는 `AskAi` 인터페이스만 알고, 실제 구현은 주입됩니다. 테스트에서는 결정적인 응답을 돌려주는 fake 구현을 사용합니다.

### 4. `models` — 3D 메타데이터

3D 모델 자체는 정적 파일(`.glb`)이지만, 그것을 *학습 단위*로 묶는 메타데이터는 도메인 객체입니다. 이 분리를 명확히 하지 않으면 파일 시스템 접근이 도메인 로직에 침투합니다.

`models` 도메인은 메타데이터만 다루고, 정적 파일은 별도 라우트가 처리하도록 경계를 그었습니다.

### 5. `exam` — 가장 복잡한 도메인

출제 / 제출 / 채점이 한 도메인 안에서 모두 일어납니다. 특히 채점 규칙은 **외부 의존성이 전혀 없는 순수 함수**로 구현되어, 단위 테스트가 가장 강력하게 작동하는 도메인입니다.

이 도메인을 마지막에 가까운 순서에 둔 이유는, 이전 도메인들에서 4-layer 패턴이 충분히 다듬어진 뒤에 가장 무게중심이 큰 도메인을 다루기 위해서였습니다.

### 6. `workflow / File (multipart)` — 진행 중

남은 도메인입니다. `multipart/form-data` 처리, 파일 저장소 추상화, 업로드 트랜잭션이 함께 다뤄져야 하기 때문에 가장 어렵게 남겨두었습니다.

## P0 이슈 처리 분포

이관 과정에서 P0(차단 수준) 이슈를 4건 식별했고, 다음과 같이 분배해 처리했습니다.

| 처리 방식 | 건수 | 의미 |
| --- | --- | --- |
| 근본 수정(Root-fixed) | 1 | 구조 결함을 직접 제거 |
| 부분 완화(Partial) | 1 | 영향 범위를 제한하고 추적 가능 상태로 전환 |
| 의도적 보류(Deferred) | 2 | 다음 마이그레이션 라운드까지 의식적으로 남김 |

모든 P0를 즉시 닫는 것이 항상 최선은 아닙니다. **닫지 않기로 결정한 P0**는 그 결정 자체가 문서화되어야 하며, 그래야 다음 작업자가 같은 항목을 다시 발견하느라 시간을 쓰지 않습니다.

## 위임 + 리뷰 루프

이관 작업의 일부는 AI 보조 도구(Claude Code)에 위임하는 방식으로 진행되었습니다. 이 워크플로우에서 중요한 것은 *위임 자체*가 아니라 **위임 결과를 사람이 다시 검토하는 단계**입니다.

이 검토 단계에서 다음과 같은 트랩 3건을 직접 검출하고 정정했습니다.

- 잘못된 추상화로 인한 의존 방향 위반
- 테스트는 통과하지만 의도와 다른 동작이 들어간 구현
- 표면적으로는 동일하지만 도메인 의미가 달라진 매핑

이 기록을 남기는 이유는, **AI 보조 작업의 안전망은 결국 사람의 검토**라는 사실을 부정하지 않기 위해서입니다. 위임은 속도를 만들지만, 검토는 정확성을 만듭니다.
