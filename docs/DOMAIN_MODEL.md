# Domain Model

이 문서는 SIMVEX 백엔드에 존재하는 5개 도메인의 *경계와 책임*을 정리합니다.
다른 문서들이 *어떻게* 와 *왜* 를 다룬다면, 이 문서는 *무엇* 을 다룹니다.

<br />

## 도메인 식별 원칙

| 기준 | 질문 |
| --- | --- |
| **라이프사이클** | 두 객체가 같은 생애주기를 공유하는가 |
| **변경 이유** | 한쪽이 바뀔 때 다른 쪽도 반드시 바뀌는가 |
| **외부 의존** | 두 객체가 같은 외부 시스템에 의존하는가 |

세 질문 중 *하나라도 "아니오"* 라면 도메인을 분리합니다. 한 도메인에 너무 많은 책임이 들어오면 Service 계층이 비대해지고, 변경 시 영향 범위를 예측하기 어려워지기 때문입니다.

<br />

## 도메인 개요

| 도메인 | 핵심 책임 | 상태 |
| --- | --- | --- |
| `memos` | 학습 메모의 CRUD | Applied |
| `workflow / Node` | 학습 흐름의 노드 관리 | Applied |
| `workflow / Connection` | 노드 사이의 연결 관리 | Applied |
| `workflow / File` | 워크플로우에 첨부되는 파일 관리 | In progress |
| `ai` | AI 질문 / 응답과 사용자별 히스토리 | Applied |
| `models` | 3D 모델 메타데이터 관리 | Applied |
| `exam` | 시험 출제 · 제출 · 채점 | Applied |

<br />

## memos

### 핵심 책임

사용자가 학습 도중 작성한 메모를 생성 · 조회 · 수정 · 삭제합니다. 가장 단순한 CRUD 도메인이며, 4-layer 패턴의 기준 사례로 사용됩니다.

### 도메인 객체

- `Memo` — 작성자, 본문, 타임스탬프, 연관 컨텍스트(예: 어떤 모델/시험에 대한 메모인지)

### Service의 주요 동작

- 메모 생성 (작성자 권한 부여)
- 메모 조회 (소유자만 접근 가능)
- 메모 수정 (소유자 검증 후 갱신)
- 메모 삭제 (소유자 검증 후 제거)

### 외부 의존

없음. 오직 Repository만 사용합니다.

### 다른 도메인과의 관계

다른 도메인을 *참조* 할 수는 있지만, 그 도메인의 내부 객체를 다루지는 않습니다. 예를 들어 메모가 특정 시험에 연결되어 있더라도, `memos` 도메인은 시험 객체의 내부 구조를 모릅니다.

<br />

## workflow

워크플로우는 단일 도메인이 아니라 **세 개의 분리된 도메인**으로 구성됩니다. 이 분리가 SIMVEX 도메인 설계에서 가장 의도적인 선택이었습니다.

### 왜 셋으로 쪼개었는가

| 분리 기준 | 결과 |
| --- | --- |
| 라이프사이클이 다름 | Node는 단독 존재 가능, Connection은 두 노드 사이에만 존재, File은 외부 저장소와 동기화 |
| 변경 이유가 다름 | 노드 모델이 바뀌어도 파일 업로드 흐름은 영향받지 않음 |
| 외부 의존이 다름 | File은 multipart 파싱과 저장소 추상화에 의존, 다른 둘은 그렇지 않음 |

셋을 한 도메인으로 묶으면 노드 삭제 로직에 연결 정리와 파일 정리 부수효과가 모두 박힙니다. 분리하면 각자의 책임이 명확해지고, 변경 시 영향 범위가 줄어듭니다.

### workflow / Node

**도메인 객체** — `Node`(워크플로우 내 학습 단위, 위치, 메타데이터)

**Service의 주요 동작**
- 노드 생성 / 조회 / 갱신 / 삭제
- 노드 삭제 시 *Connection 도메인에 정리 위임* — 직접 연결을 삭제하지 않음

### workflow / Connection

**도메인 객체** — `Connection`(시작 노드, 끝 노드, 연결 메타데이터)

**Service의 주요 동작**
- 연결 생성 (두 노드의 존재 검증)
- 연결 조회 / 삭제
- 특정 노드가 삭제될 때 관련 연결을 일괄 정리

Connection은 항상 두 Node 사이에만 존재하지만, Node 도메인의 내부 객체를 다루지는 않습니다. Node의 ID만으로 동작합니다.

### workflow / File

**도메인 객체** — `WorkflowFile`(파일 메타데이터, 저장소 경로, 연관 노드 ID)

**Service의 주요 동작**
- 파일 업로드 (multipart 파싱 → 저장소 기록 → 메타데이터 영속화)
- 파일 조회 / 삭제
- 노드 삭제 시 관련 파일 정리

**외부 의존**
- multipart 파싱
- 파일 저장소 (현재는 로컬 파일 시스템, 인터페이스로 격리되어 있음)

이 도메인이 가장 늦게 이관되는 이유는, multipart 처리와 저장소 추상화라는 추가 외부 의존이 함께 다루어져야 하기 때문입니다.

## ai

### 핵심 책임

부품 / 조립 컨텍스트에 기반한 질문 / 응답을 처리하고, 사용자별 대화 히스토리를 관리합니다.

### 도메인 객체

- `Conversation` — 사용자별 대화 세션
- `Message` — 사용자 질문 또는 AI 응답 단위

### Service의 주요 동작

- 새 대화 시작
- 질문 전송 → LLM 호출 → 응답 저장
- 사용자별 대화 히스토리 조회

### 외부 의존

LLM API. 이 의존성은 **`AskAi` 인터페이스 뒤로 격리**되어 있어 Service는 실제 LLM이 무엇인지 모릅니다.

테스트에서는 결정적인 응답을 돌려주는 fake 구현을 사용합니다. 이 격리가 없었다면 단위 테스트가 외부 API 호출에 의존하게 되어 사실상 통합 테스트로 전락했을 것입니다.

<br />

## models

### 핵심 책임

3D 모델의 **메타데이터** 를 관리합니다. 정적 파일(`.glb`) 자체는 이 도메인에서 다루지 않습니다.

### 도메인 객체

- `ModelMeta` — 모델 식별자, 표시 이름, 설명, 정적 파일 경로 참조

### Service의 주요 동작

- 모델 등록 / 갱신 / 삭제
- 모델 목록 조회
- 단일 모델 조회

### 외부 의존

없음. 정적 파일 자체는 별도 라우트가 처리하고, `models` 도메인은 메타데이터만 다룹니다.

### 분리의 근거

3D 모델 정적 파일과 메타데이터를 한 도메인에 묶으면, 메타데이터 수정 같은 가벼운 작업에 파일 시스템 접근이 끌려들어옵니다. 분리하면 메타데이터는 순수 데이터 작업으로 다룰 수 있고, 정적 파일은 별도 캐시 / 전송 전략을 가질 수 있습니다.

<br />

## exam

### 핵심 책임

시험을 출제하고, 사용자 제출을 받고, 채점합니다. 5개 도메인 중 **도메인 규칙이 가장 복잡한 도메인** 입니다.

### 도메인 객체

- `Exam` — 시험 자체 (제목, 출제 항목 묶음)
- `Question` — 개별 문항
- `Submission` — 사용자가 제출한 응답
- `Score` — 채점 결과

### Service의 주요 동작

- 시험 출제 (문항 묶음 생성)
- 사용자 제출 접수
- 제출에 대한 채점 수행
- 채점 결과 조회

### 외부 의존

없음. 특히 채점 규칙은 외부 호출 없는 **순수 함수** 로 구현되어, 단위 테스트가 가장 강력하게 작동하는 영역입니다.

### Service 무게중심

`exam`은 다른 도메인에 비해 Service 계층이 두껍습니다. 채점 규칙, 문항 검증, 제출 유효성 검사가 모두 Service에 모여 있기 때문입니다. 4-layer 구조에서 Service가 가장 두꺼울 수 있다는 것을 보여주는 사례입니다.

<br />

## 도메인이 공유하지 않는 것들

명시적으로 *공유하지 않기로 결정한* 것들입니다.

### 데이터베이스 트랜잭션

서로 다른 도메인의 Service는 *같은 트랜잭션을 공유하지 않습니다*. 한 트랜잭션 안에서 여러 도메인을 동시에 갱신하는 일이 필요하다면, 그것은 Service 위쪽 — Controller나 별도 application 계층 — 의 책임입니다.

### 도메인 객체

한 도메인의 도메인 객체가 다른 도메인의 Service에 전달되지 않습니다. 도메인 사이의 연결은 **ID 참조** 로만 표현됩니다. 예를 들어 `memos`가 특정 시험에 연결되어 있다면, 메모는 시험 ID만 가질 뿐 `Exam` 객체 자체를 가지지 않습니다.

### 영속성 스키마

각 도메인은 자기 테이블 / 키에만 접근합니다. 한 도메인의 Repository가 다른 도메인의 테이블을 직접 읽으면, 그 순간 도메인 경계가 무너집니다.

<br />

## 트레이드오프

### ID 참조로 인한 별도 조회

도메인 객체를 공유하지 않기 때문에, 화면에 *메모와 그 메모가 연결된 시험 정보* 를 함께 보여줘야 한다면 두 도메인을 각각 호출해야 합니다. 호출 횟수가 늘어나는 비용이 있습니다.

이 비용을 감수한 이유는, *경계를 흐리지 않는 것* 이 장기적으로 변경 비용을 더 크게 줄여주기 때문입니다.

### 작은 도메인의 보일러플레이트

`memos`처럼 책임이 단순한 도메인도 4-layer 전체를 따릅니다. 메모 하나 추가하기 위해 Route / Controller / Service / Repository 모두를 손봐야 합니다.

이 비용은 `ARCHITECTURE.md`의 트레이드오프 섹션에서도 같은 결론을 공유합니다 — 단순한 도메인일수록 보일러플레이트 비율이 커지지만, *기준 사례* 의 가치가 그 비용보다 큽니다.

### workflow 셋의 동조화

`workflow / Node`, `Connection`, `File` 셋은 분리되어 있지만, 노드 삭제 같은 작업에서 서로 협조해야 합니다. 분리의 대가로 *도메인 사이의 협력 흐름* 을 명시적으로 설계해야 했고, 그 흐름은 Service 위쪽에서 조율됩니다.
