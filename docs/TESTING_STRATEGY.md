# Testing Strategy

이 문서는 SIMVEX 백엔드의 테스트가 어떤 계층으로 구성되어 있고, 각 계층이 무엇을 보장하며, 무엇을 보장하지 않는지를 정리합니다.
*어떤 테스트를 쓰지 않기로 했는가*도 함께 기록되어 있습니다. 보장 범위를 명확히 하는 것이 테스트 신뢰의 출발이기 때문입니다.

<br />

## 테스트 도구

| 항목 | 선택 |
| --- | --- |
| 러너 | `node:test` (Node.js 내장) |
| 어서션 | `node:assert/strict` |
| 모킹 전략 | mock 대신 `memory` Repository 드라이버 사용 |
| 외부 의존 격리 | LLM 호출 등은 fake 인터페이스 구현으로 대체 |
<br />

## 세 가지 테스트 계층


| 계층 | 대상 | 사용 드라이버 | 외부 의존 |
| --- | --- | --- | --- |
| **단위 테스트** | Service 계층 | `memory` | 없음 |
| **통합 테스트** | Controller → Service → Repository | `memory` | 없음 |
| **PostgreSQL 통합 테스트** | Repository → SQL | `postgres` | DB 필요 |

SIMVEX의 테스트는 *대상 범위*와 *외부 의존*을 기준으로 세 계층으로 나뉩니다.

### 단위 테스트 — Service의 도메인 규칙

Service 계층은 외부 의존성을 모르므로, 단위 테스트는 가장 단순합니다. Repository 인터페이스에 `memory` 드라이버를 주입한 뒤, Service의 메서드를 직접 호출하고 결과와 부수효과를 검증합니다.

이 계층에서 확인하는 것은 **도메인 규칙**입니다. 채점 규칙, 워크플로우의 노드/연결 정합성, 메모의 권한 검증 같은 항목이 여기에 해당합니다.

### 통합 테스트 — 컨트롤러 진입부터 Repository까지

Fastify 라우트에 가까운 시작점에서 Controller → Service → Repository까지의 흐름을 한 번에 검증합니다. 여기서도 `memory` 드라이버를 사용합니다.

이 계층에서 확인하는 것은 **계층 간 연결**입니다. 라우트 스키마가 도메인 입력 타입과 일치하는지, Controller가 결과를 올바른 응답 형태로 매핑하는지, 잘못된 요청이 적절한 HTTP 응답으로 거절되는지를 함께 봅니다.

이 계층의 가장 큰 효과는 **외부 DB 없이도 통합 흐름의 회귀를 잡아낸다**는 것입니다. CI가 PostgreSQL을 띄우지 않아도 Controller / Service / Repository 계층의 결합 오류를 발견할 수 있습니다.

### PostgreSQL 통합 테스트 — SQL 검증

`postgres` 드라이버를 사용하는 통합 테스트는 별도 명령으로 분리되어 있습니다.

```bash
npm run test:postgres -w @simvex/api
```

이 계층의 목적은 **SQL이 의도대로 작성되었는가**를 검증하는 것입니다. raw `pg` 클라이언트로 SQL을 직접 실행하기 때문에, ORM이 숨겨주던 실수가 그대로 노출됩니다. 컬럼 누락, 매개변수 순서, 트랜잭션 격리 수준, 인덱스 사용 같은 항목이 여기서 발견됩니다.

<br />

## 통합 테스트 11건의 분배

현재 그린 상태의 통합 테스트는 11건이며, 4-layer가 Applied 상태인 5개 도메인에 걸쳐 분포되어 있습니다.

| 도메인 | 테스트 초점 |
| --- | --- |
| `memos` | CRUD 흐름과 권한 경계 |
| `workflow / Node · Connection` | 노드 삭제 시 연결의 정합성 |
| `ai` | fake LLM 응답 위에서의 히스토리 적재 |
| `models` | 메타데이터 등록과 조회 |
| `exam` | 출제 → 제출 → 채점 시퀀스 |

11이라는 숫자는 *충분히 많다*는 의미보다 **각 도메인의 가장 위험한 흐름을 한 번씩은 통과시킨다**는 의미에 가깝습니다. 테스트 개수를 늘리는 것보다 *어떤 흐름을 보호하고 있는가*를 명확히 하는 것이 우선이었습니다.

<br />

## 왜 mock이 아니라 memory 드라이버인가

테스트에서 외부 의존성을 격리하는 흔한 방법은 mock 객체입니다. SIMVEX는 mock 대신 `memory` Repository 드라이버를 사용합니다. 그 이유는 세 가지입니다.

### 1. mock은 인터페이스만 만족하고 의미를 흉내내지 않음

mock은 "이 메서드가 호출되면 이 값을 돌려준다"는 약속만 합니다. 실제 데이터 구조 위에서 동작하지 않기 때문에, **사용 오류가 mock에서는 통과하지만 실제 구현에서는 실패**하는 경우가 생깁니다.

memory 드라이버는 `Map` 같은 실제 데이터 구조 위에서 동작하므로, 같은 ID를 두 번 삽입하거나 존재하지 않는 ID로 조회하는 사용 오류가 그대로 드러납니다.

### 2. mock 설정 코드가 테스트의 의도를 가립니다

mock 객체는 보통 *어떤 호출이 어떤 응답을 반환하는가*를 일일이 명세해야 합니다. 테스트의 본질이 *도메인 동작 검증*인데, 화면의 절반이 mock 설정 코드라면 가독성이 크게 떨어집니다.

memory 드라이버는 사전 설정이 필요 없습니다. 초기 데이터를 직접 넣고, Service를 호출한 뒤, 상태를 확인하면 됩니다.

### 3. 통합 테스트와 단위 테스트가 같은 도구를 공유합니다

memory 드라이버는 단위 테스트뿐 아니라 통합 테스트에서도 그대로 사용됩니다. 이로 인해 두 계층이 *같은 기반* 위에 서고, 테스트 사이의 인지 부담이 줄어듭니다.

<br />

## 테스트하지 않는 것

### memory 드라이버 위에서의 동시성

memory 드라이버는 동시 쓰기 상황에서 race condition을 그대로 노출합니다. PostgreSQL의 트랜잭션 격리와는 의미가 다릅니다.

따라서 **동시성에 민감한 시나리오는 PostgreSQL 통합 테스트에서만 검증**해야 합니다. memory 드라이버에서의 그린은 동시성 안전을 보장하지 않습니다.

### 실제 LLM 응답

AI 도메인의 테스트는 fake LLM 인터페이스 구현을 사용합니다. *실제 LLM이 합리적인 응답을 주는가*는 테스트의 대상이 아닙니다. 검증하는 것은 LLM의 응답이 도메인 안에서 어떻게 흐르는가입니다.

### 프론트엔드 통합

`apps/web`은 별도 타입 체크(`npm run typecheck -w @simvex/web`)로 검증되며, 백엔드 통합 테스트의 대상이 아닙니다. 두 앱 사이의 계약은 `@simvex/shared` 패키지의 타입으로 일급 보호됩니다.

## CI 실행 순서

빠른 피드백을 위해 외부 의존이 적은 순서대로 실행합니다.

1. 타입 체크 — `npm run typecheck -w @simvex/api`, `npm run typecheck -w @simvex/web`
2. 단위 + 통합 테스트 (memory 기반) — `npm run test:unit -w @simvex/api`
3. PostgreSQL 통합 테스트 — `npm run test:postgres -w @simvex/api`
4. 마이그레이션 검증 — `npm run migrate:validate -w @simvex/api`
5. DB 스모크 체크 — `npm run db:smoke -w @simvex/api`

타입 체크와 단위 / 통합 테스트는 외부 의존이 없기 때문에, CI 진입 직후 가장 먼저 통과 여부가 드러납니다.

<br />

## 트레이드오프

### memory 드라이버의 충실도 한계

memory 드라이버는 *드라이버 인터페이스의 의미를 흉내내는 진짜 구현*이지만, PostgreSQL의 모든 의미를 흉내내지는 않습니다. 트랜잭션, 외래 키 제약, 격리 수준 같은 항목은 PostgreSQL 통합 테스트에서만 검증됩니다.

이 한계를 받아들이는 대신, **빠른 피드백 루프**를 얻었습니다.

### PostgreSQL 통합 테스트의 무게

`postgres` 드라이버를 사용하는 통합 테스트는 DB 기동 시간을 포함하므로, 로컬에서 자주 돌리기에는 부담이 있습니다. 그래서 평소 개발 사이클에서는 memory 기반 테스트만 반복하고, PostgreSQL 테스트는 *큰 변경이 끝났을 때*나 *CI*에서만 실행합니다.

### 통합 테스트 11건의 범위

각 도메인의 가장 위험한 흐름은 보호하지만, *모든 분기*를 통합 테스트가 덮지는 않습니다. 분기 단위의 세부 검증은 Service 단위 테스트의 책임입니다.
